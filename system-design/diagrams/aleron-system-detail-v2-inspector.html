System objective

The system turns each member’s diagnostic budget into the clearest honest read we can give on the next best thing to do for them, and how sure we are of it. Risk-reduction benefit is measured in QALYs, the quality-adjusted life-years gained against this member’s current care. Vitality benefit is measured against the four felt outcome dimensions. The next best thing is meant broadly. It can be a treatment, a change in habit, a screening test, ongoing monitoring, doing nothing for now, or simply collecting more information first, like ordering a calcium scan.

The diagnostic budget is what we spend to measure one member over a year. It is set per member because that is how the economics work. What we spend measuring someone has to fit inside the fee they pay. The budget is only a fraction of the roughly $4,500 annual membership, which also has to cover the clinic’s costs and margin. We do not commit all of it up front; a practice-wide float stays in reserve, so we can spend more on a member when a test would change more for them.

The budget bounds what we can measure; it does not choose what to measure. That choice comes from the objective and from value of information, meaning how much a given test would move a decision. We order a test only when its result would change a decision by more than the test costs. Because members are mostly healthy, baseline risks are low and the gain from any single action is small, so that bar stays high.

For each member the system gives the physician an action map that ranks optional actions by their relevant value axis and how sure we are, a must-do channel carrying the steps that are required regardless, and — drafted from those — the note and its orders, unsigned. The physician dispositions that draft: accept, reject, modify, or add something off-library. Their signature is the gate. It locks the note into the record, releases each order through its own channel, and keeps prescription sign-off native. Care executes, and what comes back — results, patient reports, outside records — re-enters the patient data model and starts the next pass. We own the diagnostic spend and lay out the options; the member decides which treatments to pursue and pays for those separately.

All of this answers to one rule it cannot override. We report only what we actually know and how well we know it, and we never treat a prediction as proof.

PartWhat it’s forHow we know it’s working
Data buyer Spend the diagnostic budget to gather this member’s data. It draws on patient-reported answers, passive wearable signals, and paid tests, and it decides what is worth buying. It buys a test only when the result would change a decision by more than the test costs, and the total stays inside the member’s budget and float.
Patient data model Hold the usable structured profile for the member: reported answers, wearable signals, paid test results, derived fields, and provenance. Downstream models read the same structured profile rather than each inventing its own version of the member.
Models Run the model family. Risk models estimate disease probabilities and modifiable risk drivers; the vitality model reads felt outcomes, associated drivers, gates, and trends. Whether a model finding becomes an action is left to the action library. Risk outputs calibrate against future events; vitality outputs calibrate against within-person felt outcomes and driver response.
Action library Hold the canonical catalog of diagnostics, therapeutics, lifestyle changes, procedures, monitoring, and rules. It decides which known actions are eligible for this member and what each action could change. Every library entry has enough structure to classify, gate, or score it without prose judgment.
Scorer + classifier Act as the shared arbitration layer outside the deterministic and inference modules. It combines library candidates, provisional inference packets, patient data, model outputs, and guideline rules to rank optional actions, route must-dos, gate uncertain items, and block contraindicated items. Optional opportunities get scored on the relevant value axis; required or contraindicated items are classified rather than forced onto the QALY scale.
Inference module Use patient data plus action-library coverage to find discordance, gaps, and unmodeled patterns. It checks whether the library already covers the pattern, then turns only uncovered patterns into provisional candidate packets. It suppresses duplicates and does not score its own output. Every useful discovery returns to the scorer and classifier as a structured candidate packet.
Action map Compare every optional action on the relevant value axis and the strength of the evidence, then pick the feasible bundle that fits this person. A big-but-unsure option and a small-but-sure one land apart, and no other feasible bundle helps more once overlap and the must-dos are counted.
Must-dos Carry the steps that are required regardless: a referral forced by a very high model risk, a red flag from the inference module, or a standing guideline like an overdue screen. The channel lands in the draft note as obligations; it never ranks on the action map. Nothing required slips through, a model breach or a red flag becomes a referral, and the mandate comes from the gate, not from a model.
AI-drafted note & orders Reduce the final typed state into the unsigned draft: a narrative assessment plus the staged order set, with the required channel inside it and provenance on every line. A care plan is this note before its signature. It reads only the final state, adds no unscored recommendations, and passes synthesis QA before display. Nothing executes from this node.
Physician note & orders The only human-owned node. The physician dispositions the draft — accept, reject, modify, add off-library — and signs. The signature locks the note into the record and releases each order through its own channel; prescription sign-off stays native. Nothing in the system executes without it. Rejected levers are suppressed next cycle; physician-added gaps are logged for library review.
Patient action Care executes: draws happen, prescriptions dispense, referrals are seen, goals are or are not done. Execution is information — what happens and what does not both come back.
New information Collect what returns after care: results against orders, patient reports and messages including disagreement, wearable changes, and outside records. These update the patient data model first. Every return path is the same kind of event — re-entry at the top of the pipeline before any model reruns.

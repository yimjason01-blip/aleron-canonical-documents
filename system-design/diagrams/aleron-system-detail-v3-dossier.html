Target-state architecture with partial contract implementation. Current operational capability covers packet transport, linkage checks, fail-closed analysis status, physician review, and synthetic staging release. Clinical Risk and Vitality execution, diagnostic-budget optimization, production action ranking, inference discovery, AI synthesis, and automatic closed-loop reruns are planned or prototype-only. Tiles describe ownership boundaries and intended contracts, not an inventory of deployed services.
System objective

The system turns each member’s diagnostic budget into the clearest honest read we can give on the next best thing to do for them, and how sure we are of it. Risk-reduction benefit is measured in QALYs, the quality-adjusted life-years gained against this member’s current care. Vitality benefit is measured against the four felt outcome dimensions. The next best thing is meant broadly. It can be a treatment, a change in habit, a screening test, ongoing monitoring, doing nothing for now, or simply collecting more information first, like ordering a calcium scan.

The diagnostic budget is what we spend to measure one member over a year. It is set per member because that is how the economics work. What we spend measuring someone has to fit inside the fee they pay. The budget is only a fraction of the roughly $4,500 annual membership, which also has to cover the clinic’s costs and margin. We do not commit all of it up front; a practice-wide float stays in reserve, so we can spend more on a member when a test would change more for them.

The budget bounds what we can measure; it does not choose what to measure. That choice comes from the objective and from value of information, meaning how much a given test would move a decision. We order a test only when its result would change a decision by more than the test costs. Because members are mostly healthy, baseline risks are low and the gain from any single action is small, so that bar stays high.

For each member we give the physician three things. An action map ranks optional actions by their relevant value axis and how sure we are. A separate must-do list carries the steps that are required regardless, like an overdue screen or a referral forced by a very high risk. An AI clinical summary reduces the final state into a concise clinical frame, without adding new recommendations. We own the diagnostic spend and lay out the options; the member decides which treatments to pursue and pays for those separately.

All of this answers to one rule it cannot override. We report only what we actually know and how well we know it, and we never treat a prediction as proof.

Each part commits to exactly the logic it needs, enough to be clear without so much that a slip can hide. One job, one check.
PartWhat it’s forHow we know it’s working
Data buyer Spend the diagnostic budget to gather this member’s data. It draws on patient-reported answers, passive wearable signals, and paid tests, and it decides what is worth buying. It buys a test only when the result would change a decision by more than the test costs, and the total stays inside the member’s budget and float.
Patient data model Hold the usable structured profile for the member: reported answers, wearable signals, paid test results, derived fields, and provenance. Downstream models read the same structured profile rather than each inventing its own version of the member.
Models Run the model family. Risk models estimate disease probabilities and modifiable risk drivers; the vitality model reads felt outcomes, associated drivers, gates, and trends. Whether a model finding becomes an action is left to the action library. Risk outputs calibrate against future events; vitality outputs calibrate against within-person felt outcomes and driver response.
Action library Hold the canonical catalog of diagnostics, therapeutics, lifestyle changes, procedures, monitoring, and rules. It separates first-order actions, diagnostic/Get Evidence items, classifier routes, second-order actions, lifestyle prescriptions, procedure thresholds, and surrogate or enablement items. Every entry has enough structure to classify, gate, score, or hold it without prose judgment, and no second-order item is forced onto the ranked QALY map.
Scorer + classifier Act as the shared arbitration layer outside the deterministic and inference modules. It combines library candidates, provisional inference packets, patient data, model outputs, and guideline rules to rank first-order optional actions, route classifiers, hold second-order actions, surface Get Evidence items, route must-dos, and block contraindicated items. Optional first-order opportunities get scored on the relevant value axis; diagnostics use VOI; classifiers and second-order items are routed or held rather than forced onto the QALY scale.
Inference module Use patient data plus action-library coverage to find discordance, gaps, and unmodeled patterns. It checks whether the library already covers the pattern, then turns only uncovered patterns into provisional candidate packets. It suppresses duplicates and does not score its own output. Every useful discovery returns to the scorer and classifier as a structured candidate packet.
Action map Compare optional first-order actions on the relevant value axis and the strength of the evidence, then pick the feasible bundle that fits this person. A big-but-unsure option and a small-but-sure one land apart, no classifier or second-order hold masquerades as a computed action, and no other feasible bundle helps more once overlap and must-dos are counted.
Must-dos Surface the steps that are required regardless: a referral forced by a very high model risk, a red flag from the inference module, or a standing guideline like an overdue screen. This is parallel to the action map, not an input into it. Nothing required slips through, a model breach or a red flag becomes a referral, and the mandate comes from the gate, not from a model.
AI summary Reduce the final typed state into a concise physician-facing synthesis: what matters most, what the few next steps are, and why salient alternatives were deferred. It reads only the final state, adds no unscored recommendations, preserves must-dos separately, and passes synthesis QA before display.
New information Capture new observations after the plan: outcomes, follow-up tests, wearable changes, patient updates, and adherence signals. These update the patient data model first. New facts enter the normalized profile before any deterministic model or inference module reads them again.
